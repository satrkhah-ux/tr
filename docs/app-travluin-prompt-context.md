# App Travluin - ملف سياق مختصر واحترافي لكتابة برومبتات تطوير النظام

## 1. الهدف من هذا الملف

هذا الملف مخصص لإرساله إلى ChatGPT أو أي مساعد ذكاء اصطناعي قبل طلب كتابة برومبتات تطوير النظام.

الهدف هو أن يفهم المساعد:

- ما هو نظام App Travluin.
- ما أصل النظام ولماذا تم استنساخه.
- ما هي البنية التقنية الحالية.
- ما هي أهم الصفحات والوظائف.
- ما هي طريقة كتابة برومبت احترافي قليل التوكن وعالي الإنتاجية.
- كيف يحول أي مشكلة أو فكرة تطوير إلى طلب تقني واضح قابل للتنفيذ داخل المشروع.

استخدم هذا الملف كـ "سياق ثابت" ثم أضف بعده المشكلة المطلوبة أو الميزة الجديدة.

---

## 2. تعريف النظام

App Travluin هو نظام إدارة وتشغيل لشركة سياحية، مبني كنسخة مستنسخة ومطورة من النظام الأصلي الخاص بإدارة عمليات Traveliun.

النظام الأصلي يستخدم لإدارة:

- العملاء.
- الموظفين.
- الصلاحيات.
- البكجات والعروض السياحية.
- الدول والمدن.
- الفنادق والغرف.
- الطيران.
- الرحلات البحرية.
- المواصلات.
- الخدمات.
- التأشيرات.
- الشروط والأحكام.
- إنشاء عروض العملاء.
- إنتاج ملفات PDF وروابط عروض للعملاء.

الغرض من الاستنساخ ليس مجرد نسخ الواجهة، بل بناء نسخة حديثة قابلة للفهم والتطوير بواسطة الذكاء الاصطناعي، ثم تحويلها إلى نظام أكثر ذكاءً وكفاءة.

## 3. الهدف التجاري من المشروع

تحويل النظام من لوحة إدارة تقليدية إلى منصة تشغيل سياحي ذكية تساعد الشركة على:

- تسريع إنتاج العروض السياحية.
- تقليل الأخطاء اليدوية في البرامج والأسعار والليالي.
- توحيد شكل ملفات PDF وروابط العملاء.
- رفع إنتاجية الموظفين.
- إدارة العملاء والموظفين والصلاحيات بوضوح.
- إنشاء برامج سياحية من عروض محفوظة أو من عروض مخصصة.
- تجهيز النظام لاحقًا للربط مع API للفنادق والطيران والرسائل والدفع.
- استخدام الذكاء الاصطناعي لاقتراح برامج، مراجعة الأخطاء، وتحسين النصوص.

## 4. التقنية الحالية

المشروع مبني على:

- Next.js 16 App Router.
- React 19.
- TypeScript strict.
- Tailwind CSS v4.
- Lucide React للأيقونات.
- shadcn/ui كأساس واجهات.
- بنية صفحات داخل `src/app`.
- مكونات Traveliun داخل `src/components/traveliun`.
- بيانات مبدئية ومولدة داخل `src/lib`.

ملاحظات مهمة:

- هذه نسخة Next.js 16 وليست مثل الإصدارات القديمة.
- يجب قراءة التوثيق المحلي داخل `node_modules/next/dist/docs/` قبل كتابة كود متعلق بـ Next.js.
- يجب الالتزام بـ TypeScript strict وعدم استخدام `any`.
- يجب استخدام نمط المشروع الحالي وعدم إدخال مكتبات كثيرة دون ضرورة.
- يجب الحفاظ على اتجاه RTL والتجربة العربية.

## 9. مولد البكجات السياحية

مولد البكجات هو شاشة واحدة تعمل بمراحل منسدلة لتقليل تشتت الموظف.

المراحل الحالية:

1. بيانات العميل والرحلة.
2. البحث عن برنامج سابق.
3. المدن والفنادق.
4. الطيران.
5. الخدمات.
6. الشروط.
7. المعاينة والإنتاج.

### المرحلة الأولى: بيانات العميل والرحلة

المدخلات:

- اسم العميل.
- رقم هاتف العميل.
- الدولة.
- تاريخ الوصول.
- تاريخ المغادرة.
- عدد الأيام.
- عدد البالغين أكثر من 12 سنة.
- عدد الأطفال أقل من 12 سنة.
- عدد الأطفال أقل من 6 سنوات.

ملاحظة مهمة:

لا يظهر "تطابق الليالي" في هذه المرحلة. تطابق الليالي يظهر فقط عند إنشاء البرنامج واختيار المدن والفنادق.

### مرحلة البحث عن برنامج سابق

بعد الضغط على التالي:

- تغلق بيانات العميل.
- يتم البحث عن برامج محفوظة بنفس الدولة ونفس عدد الأيام.
- لا يشترط نفس تاريخ الوصول أو المغادرة.
- إذا كان عدد الأشخاص مطابقًا يظهر البرنامج في أول الخيارات.
- إذا لم يكن مطابقًا يظهر كمقترح قابل للنسخ والتعديل.

### مرحلة المدن والفنادق

الموظف يختار:

- المدينة الأولى.
- عدد الليالي.
- الفندق.

ثم يكرر ذلك حتى يغطي عدد الليالي.

هنا فقط يظهر تطابق الليالي:

- مكتمل.
- ليالي متبقية.
- ليالي زائدة.

### مرحلة الطيران

فكرتها:

- اختيار الرحلات.
- الأسعار.
- الأمتعة.
- إمكانية الربط لاحقًا بـ API للطيران.

### مرحلة الخدمات

اختيار الخدمات المشمولة مثل:

- استقبال وتوديع.
- جولات سياحية.
- شريحة اتصال.
- خدمة عملاء.
- سيارة خاصة.

### مرحلة الشروط

إضافة أو تعديل شروط العرض.

### مرحلة المعاينة والإنتاج

المخرجات المطلوبة:

- PDF.
- صورة.
- ملف مكتوب.
- رابط عميل.

## 20. قواعد مهمة لأي تطوير

- لا يتم تغيير التصميم جذريًا بدون سبب.
- لا يتم تحويل النظام إلى صفحة تسويقية.
- واجهات الإدارة يجب أن تكون عملية وكثيفة لكن منظمة.
- الموظف يجب أن يرى ما يحتاجه فقط في كل مرحلة.
- أي منطق معقد يجب أن يكون واضحًا وقابلًا للاختبار.
- بيانات العملاء والأسعار يجب فصلها عن واجهة العميل حسب الصلاحيات.
- روابط العملاء لا تعرض سعر الشراء أو الربح.
- ملفات PDF يجب أن تكون بهوية موحدة.
- أي ربط خارجي يجب أن يتم عبر API رسمي أو اتفاقية واضحة.
- ممنوع وضع كلمات مرور أو مفاتيح API داخل الكود.
